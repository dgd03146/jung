# Learning Reflection

이 프로젝트를 통해 배운 기술적·메타적 학습을 정리합니다.

---

## 기술적 배움

### 상태 우선(State-First) 사고법
Optimistic Updates 설계를 통해 학습:
- UI는 상태의 결과물이지, 상태가 UI를 따라가면 안 됨
- 상태를 먼저 설계하고, UI는 그 상태를 반영하는 구조

### 성능은 캐싱이 아니라 불필요한 작업 제거
- `isMutating` 체크로 중복 refetch 방지가 `staleTime`보다 효과적
- await → void prefetch로 불필요한 블로킹 제거
- content 컬럼 제거로 90% 페이로드 감소

### 타입 안전성의 가치
tRPC로 개발 속도와 안정성 모두 확보:
- API 스키마 변경 시 컴파일 타임에 에러 감지
- 런타임 에러를 컴파일 타임으로 전환
- 코드 생성 없이 E2E 타입 안전성

---

## 메타 학습 (사고 구조)

### FSD는 경계(Boundaries) 설계
- 폴더 구조가 아니라 레이어 간 의존성 방향이 핵심
- features → entities → shared
- "어디에 둘까?"보다 "어떤 의존성을 허용할까?"

### 디자인 시스템은 브랜드 언어
- UI 도구가 아니라 일관된 토큰과 패턴의 집합
- 사용자 경험의 예측 가능성을 높임
- 개발자 간 커뮤니케이션 비용 감소

### 유지보수성의 핵심: "왜"를 설명할 수 있는 구조
- 6개월 후 코드를 다시 봐도 의도를 이해할 수 있어야 함
- Trade-offs를 문서화하고, 대안을 기록
- 결정의 맥락을 코드 주석이 아닌 아키텍처로 표현

---

## Trade-offs & Next Time

### 발견한 제약과 보완책
- **Vanilla Extract의 동적 스타일 제약** → CSS Variables로 보완 가능
- **초기 FSD 구조 복잡도** → 팀 온보딩 문서 필요 (AGENTS.md로 보완)
- **테스트 커버리지 부족** → E2E 테스트 우선 확대 예정

### 다음에는 다르게 할 것
- 테스트를 기능 구현과 동시에 작성
- 성능 지표를 초기부터 모니터링
- 문서화를 "나중에"가 아닌 "지금" 작성

---

## Why This Matters

이 프로젝트는 "학습에서 실전으로"의 전환을 보여줍니다.

단순히 코드를 실행하는 것이 아니라, 설계 사고(design thinking)를 통해 기술적 호기심을 확장 가능한 사용자 중심 제품으로 전환하는 과정입니다.

### 결과
- 실제 사용자가 있는 프로덕션 환경
- 20개 feature 모듈을 일관된 구조로 관리
- 성능 지표 평균 70% 개선
- 타입 안전성 100% 유지
- 12개 optimistic mutation으로 즉각적인 UX 제공

지금도 이 프로젝트는 계속 발전 중입니다. 완벽함보다는 배움을, 결과보다는 과정을 더 중요하게 생각합니다.
